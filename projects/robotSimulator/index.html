Robot Simulation

Simulate a robot from CAD to Gazebo, and control it using Robot Operating System (ROS) Topics.

On the left, the CAD model of the robot design. On the Right, the robot in Gazebo.

Robot Design

The robot above was designed and developed by the Autonomous Robotics Club (ARC) to meet the specifications set by the Intelligent Ground Vehicle Competition (IGVC). The primary requirements included a maximum speed of five miles per hour and a load carrying capacity of twenty pounds. The design process involved extensive brainstorming and iterative revisions, ultimately resulting in the four-wheel differential drivetrain chassis that is currently showcased.

CAD

The Computer-Aided Design (CAD) of the robot was completed using Autodesk Inventor, with the model simplified into two distinct parts: the moving components and the static components. To proceed with the robot simulation, a file known as an URDF file was created from the CAD model. The URDF file can be obtained through two methods: manual definition of the components or utilizing a program. Unfortunately, the only available plugin for creating URDF files is for SolidWorks. To ensure that only the essential components were included in the URDF file, I created a new component that consolidated the frame and non-moving components into a single stationary block with identical size, center of mass, and moment of inertia. The moving components were then simplified to wheels with attached axles. These two components were saved as .step files and imported into SolidWorks for use with the plugin. Following the steps provided by the plugin wizard generated an URDF file that fully defined the physics of the robot. The detailed CAD design was then converted into STL files, which defined the visual appearance of the robot for our simulation software.

Docker & Ubuntu

In order to provide a foundation for my teammates to test their code, it was important to ensure the robot simulation was portable. To achieve this, I created a Docker image running Ubuntu 20 that includes all necessary components for running the Gazebo robot simulation software. The image includes Robot Operating System (ROS), Gazebo, NAV2, and various Python libraries, making it ready for testing and development.

URDF File

With the initial URDF file in place, I was able to further define the characteristics of the robot. I incorporated the visual aspect of the robot, utilizing the STL files, which required some adjustments to align the reference points of the components. Once this was completed, I added the necessary plugins for NAV2 to control the wheels and gather location data, allowing for a more robust and accurate simulation.

Ros Launch & File setup

In accordance with the established file system for ROS projects, I properly organized and placed the mesh models, configuration files, and other necessary files in the appropriate directory. I then utilized the Colcon build process and executed the ROS launch command to launch Gazebo and load the robot onto the default ground plane.

Controlling & Connecting to the Robot

The robot, utilizing the capabilities of Nav2 and Gazebo, was configured to host ROS topics for wheel odometry, velocity data, and a simulated six-axis Inertial Measurement Unit (IMU). By executing the cmd_vel command, I was able to transmit velocity commands to the robot, allowing for dynamic control of its movement and direction.

Summary

The implementation of a single launch file provides a convenient and efficient method for accessing a fully functional robot simulation, allowing for streamlined testing and debugging of code by multiple team members prior to physical fabrication. This approach significantly improves the development process and ensures that any potential issues are identified and resolved before the construction phase.
